# AGENTS.md

このリポジトリは、小さい Web アプリを継続的に追加して配信するための場所です。
このファイルは百科事典ではなく、エージェント向けの短い入口です。詳細は各ドキュメントに分離します。

## 最初に読む順番

1. README.md
2. repo-docs/index.md
3. ARCHITECTURE.md
4. FRONTEND.md
5. PLANS.md

## 現在の状態

- 現在はブートストラップ段階です。
- まだ配信対象のアプリ本体や技術スタックは確定していません。
- 先にドキュメント構造を整え、以後の実装を安定して進められる状態を目指します。

## 作業ルール

- このファイルは短く保つ。永続的な知識は repo-docs/ に置く。
- docs/ は GitHub Pages 用の公開物に限定し、内部向けの運用文書を置かない。
- 仕様が曖昧なまま実装を始めない。必要なら repo-docs/product-specs/ に先に整理する。
- 非自明な変更は repo-docs/exec-plans/active/ に実行計画を作ってから進める。
- コード変更と同時に、関連するドキュメントも更新する。
- 共有抽象化は重複が実際に出るまで作らない。
- 小さいアプリでは、読めること、直せること、再現できることを優先する。

## 期待するリポジトリ構造

- docs/: GitHub Pages 用の公開コンテンツ
- repo-docs/: エージェントと人間が共有する内部記録システム
- apps/: 個別の Web アプリ本体を置く想定のディレクトリ
- 必要になってから shared/ や packages/ を追加する

## 変更前提

- 新しいアプリを作るときは、まずプロダクト仕様を書く。
- 画面や UX を伴う変更は FRONTEND.md に従う。
- 構造の変更は ARCHITECTURE.md に反映する。
- 完了条件や作業順は PLANS.md に沿って管理する。

## 完了の定義

- 変更対象に対応する仕様または計画が存在する。
- ローカル実行方法が README または対象アプリの文書で追える。
- 変更フローに既知のコンソールエラーがない。
- 採用済みスタックに lint や test があるなら、それを実行して結果を残す。
