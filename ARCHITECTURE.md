# ARCHITECTURE.md

## 目的

このリポジトリは、小さい Web アプリを複数管理できる状態を目指します。
当面は、巨大なモノリスよりも、独立して理解しやすいアプリ群として育てます。

## 設計原則

- アプリ単位で独立性を保つ。
- 先に複雑な共有基盤を作らない。
- エージェントが読んで推論しやすい単純な構造を優先する。
- 技術選定は新規性より可読性と保守性を重視する。
- ルールは会話ではなく、リポジトリ内の文書と設定に残す。

## 推奨レイアウト

```text
apps/
  <app-name>/
docs/
  public site for GitHub Pages
repo-docs/
  design-docs/
  exec-plans/
  product-specs/
  references/
```

## アプリ境界

- 1 アプリ 1 明確な入口を持つ。
- 配信単位、README、実行方法はアプリごとに分離する。
- 別アプリに依存しない構成を基本とする。
- 共通コードは、2 つ以上のアプリで重複が確認されてから検討する。

## スタック選定の初期方針

- 静的な LP や単機能ツールなら、HTML/CSS/JavaScript を優先する。
- 状態管理や複数画面の対話が増えるなら、Vite ベースの TypeScript 構成を検討する。
- フレームワーク導入は、必要な複雑さが明確になってから行う。

## いま決まっていること

- まずは repo-docs-first で進める。
- 実装前に仕様と作業計画の置き場所を固定する。
- docs/ は公開物専用として空けておく。
- 今はアプリ本体より、後続のエージェント作業を安定させるための土台作りを優先する。
